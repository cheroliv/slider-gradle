== Description

`com.cheroliv.slider` is a Gradle plugin that encapsulates the full lifecycle of *AsciidoctorJ Reveal.js* slide generation.
It exposes a minimal DSL to the consumer and handles all repository, dependency, and task configuration internally.
It also integrates an AI assistant (E-3PO) for automated AsciiDoc deck generation via multiple LLM providers.

== Current version : {plugin-version}

== Prerequisites

* JDK 25 (tested with Eclipse Temurin 25.0.2)
* Gradle 9.4.0+
* Node.js / npx (for the `serveSlides` task)

== Key Technical Decisions

=== Plugin IDs â€” `.classic` suffix
Since `asciidoctor-gradle-jvm-slides:5.0.0-alpha.1`, the plugin IDs have changed:

[cols="1,1"]
|===
| Previous (4.x) | Current (5.x)

| `org.asciidoctor.jvm.gems`
| `org.asciidoctor.jvm.gems.classic`

| `org.asciidoctor.jvm.revealjs`
| `org.asciidoctor.jvm.revealjs.classic`
|===

=== Execution Mode â€” `OUT_OF_PROCESS`
`JAVA_EXEC` was removed in `5.0.0-alpha.1` due to Gradle API changes around closure serialisation.
`OUT_OF_PROCESS` is the correct replacement and works on Gradle 9 + Java 25.

=== pluginManagement repository â€” `repo.gradle.org`
The transitive dependency `grolifant:0.12` (required by `jruby-gradle-core-plugin`) is only available at:

[source]
----
https://repo.gradle.org/gradle/libs-releases/
----
This must be declared in the consumer's `pluginManagement.repositories`.

=== Ivy repository for Ruby gems
The gem `asciidoctor-revealjs` is resolved via a dedicated Ivy repository:
[source,kotlin]
----
project.repositories.ivy { repo ->
    repo.url = project.uri("https://rubygems.org/gems/")
    repo.patternLayout { layout -> layout.artifact("[module]-[revision].gem") }
    repo.metadataSources { s -> s.artifact() }
    repo.content { c -> c.includeGroup("rubygems") }
}
----
The `@gem` qualifier is mandatory:
[source,kotlin]
----
project.dependencies.add("asciidoctorGems", "rubygems:asciidoctor-revealjs:3.1.0@gem")
----

=== Scaffold â€” First-Use Auto-Initialisation
On the first Gradle task execution, `SliderManager.Scaffold` runs three independent checks
and generates missing resources automatically.

==== slides/ initialisation â€” `scaffoldSlidesIfAbsent()`
Checks whether the consumer project has a complete `slides/` directory. If not, extracts
the bundled `slides.zip` from the plugin classpath into the project directory.

A `slides/` directory is considered complete when `slides/misc/` contains:

* `deck.properties` â€” at least one `*.deck.file` key
* `index.html` â€” presentation dashboard
* at least one `*-deck.adoc` file matching a `*.deck.file` key in `deck.properties`

The bundled zip must be placed at:

[source]
----
slider/src/main/resources/slides.zip
----

Gradle embeds it automatically in the plugin jar. The zip entries must use relative paths
starting with `slides/` so extraction into `projectDirectory` produces the correct layout:

[source]
----
slides/
â””â”€â”€ misc/
    â”śâ”€â”€ example-deck.adoc
    â”śâ”€â”€ index.html
    â”śâ”€â”€ deck.properties
    â””â”€â”€ images/
----

The completeness check is implemented as a pure function `isSlidesConfigComplete(miscDir: File)`
inside `SliderManager.Scaffold` â€” testable in pure Kotlin without GradleTestKit.

==== slides-context.yml initialisation â€” `scaffoldSlidesContextIfAbsent()`
If `slides-context.yml` is absent from the consumer project directory, generates one automatically
by serialising a default `SlidesConfiguration` instance via `yamlMapper`. This guarantees the
output is structurally valid and consistent with the data model â€” no hardcoded YAML strings.

[source,kotlin]
----
val default = SlidesConfiguration(
    srcPath = "docs/asciidocRevealJs",
    pushSlides = GitPushConfiguration(
        from = "build/docs/asciidocRevealJs",
        to = "build/slides-repo",
        branch = "main",
        message = "deploy slides",
        repo = RepositoryConfiguration(
            name = "slides",
            repository = "https://github.com/your-org/your-slides-repo.git",
            credentials = RepositoryCredentials(
                username = "your-username",
                password = "your-token"
            )
        )
    ),
    ai = AiConfiguration(
        gemini = listOf("your-gemini-api-key"),
        mistral = listOf("your-mistral-api-key"),
        huggingface = listOf("your-huggingface-api-key"),
    )
)
yamlMapper.writeValue(slidesContext, default)
----

==== example-deck-context.yml initialisation â€” `scaffoldDeckContextIfAbsent()`
If `slides/misc/example-deck-context.yml` is absent, generates a ready-to-use template
by serialising a default `DeckContext` instance via `yamlMapper`.

All three functions are no-ops if the resource already exists â€” existing consumer content is never overwritten.

== SliderPlugin.apply() â€” Orchestration

[source,kotlin]
----
override fun apply(project: Project) {
    with(project) {
        checkJavaVersion()
        scaffoldSlidesIfAbsent()
        scaffoldSlidesContextIfAbsent()
        scaffoldDeckContextIfAbsent()
        configureRepositories()
        applyPlugins()
        configureDependencies()
        configureExtensions()
        registerTasks()
        afterEvaluate { plugins.apply(AssistantPlugin::class.java) }
    }
}
----

== AI Integration â€” AssistantPlugin

`AssistantPlugin` is applied automatically inside `SliderPlugin.apply()` via `afterEvaluate`.
It registers the `generateDeck` task and manages four LLM providers.

=== Supported providers

[cols="1,1,1"]
|===
| Provider | Module | Key in slides-context.yml

| Ollama (local)
| `langchain4j-ollama`
| â€” (no key required)

| Google Gemini
| `langchain4j-google-ai-gemini`
| `ai.gemini`

| Mistral AI
| `langchain4j-mistral-ai`
| `ai.mistral`

| HuggingFace (via OpenAI router)
| `langchain4j-open-ai`
| `ai.huggingface`
|===

=== generateDeck task

The provider is selected at the command line with `-Pai.provider`. Defaults to `ollama` when absent.

[source,bash]
----
# Default: ollama (local)
./gradlew generateDeck -Pdeck.context=slides/misc/my-deck-context.yml

# Gemini
./gradlew generateDeck -Pdeck.context=slides/misc/my-deck-context.yml -Pai.provider=gemini

# Mistral AI
./gradlew generateDeck -Pdeck.context=slides/misc/my-deck-context.yml -Pai.provider=mistral

# HuggingFace
./gradlew generateDeck -Pdeck.context=slides/misc/my-deck-context.yml -Pai.provider=huggingface
----

[cols="1,2,2"]
|===
| `-Pai.provider` | Default model | API key

| `ollama` _(default)_
| `smollm:135m`
| none â€” local inference

| `gemini`
| `gemini-2.5-flash`
| `ai.gemini[0]`

| `mistral`
| `mistral-small-latest`
| `ai.mistral[0]`

| `huggingface`
| `Llama-3.1-8B-Instruct:sambanova`
| `ai.huggingface[0]`
|===

The property is read via `Project.aiProvider` (constant `AssistantManager.PROP_AI_PROVIDER = "ai.provider"`).
Unknown values fall back to `ollama` with a console warning.

The task reads the `DeckContext` from the YAML file, builds a system prompt with full
AsciiDoc/Reveal.js syntax knowledge, and sends it to the resolved `ChatModel`.
The generated `.adoc` file is written to `slides/misc/` at the path defined by `outputFile`.

=== Prompt strategy

The system prompt (`deckSystemPrompt`) teaches the LLM AsciiDoc/Reveal.js syntax.
This approach is effective even for small models (1Bâ€“8B). Key rules enforced:

* Maximum 5 bullet points per slide with `[%step]`, 7 without
* Maximum 10 lines of code per `[source,...]` block
* Never combine a bullet list and a code block on the same slide
* `[NOTE.speaker]` and `[.notes]` presence controlled by `NotesConfiguration`
* `SlideHint` values guide content per slide without replacing LLM generation

== Gradle Feature Compatibility

The plugin declares its compatibility with opt-in Gradle features via the
`compatibility {}` block introduced in Plugin Publish plugin v2.1.0.
This declaration is mandatory â€” the Gradle Plugin Portal emits a warning
during `publishPlugins` if it is absent, and plans to eventually reject plugins without it.

[source,kotlin]
----
import org.gradle.plugin.compatibility.compatibility

gradlePlugin {
    plugins {
        create("slider") {
            // ...
            compatibility {
                features {
                    // asciidoctorRevealJs runs OUT_OF_PROCESS via JRuby â€”
                    // not compatible with Configuration Cache.
                    // Will be revisited when asciidoctor-gradle stabilises
                    // beyond 5.0.0-alpha.1.
                    configurationCache = false
                }
            }
        }
    }
}
----

Declaring `configurationCache = false` is honest and recommended â€” it informs
users clearly and has no negative consequences for the plugin's Portal ranking
beyond the Configuration Cache badge.
