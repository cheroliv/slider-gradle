== Description

`com.cheroliv.slider` est un plugin Gradle qui encapsule l'intégralité du cycle de vie de génération de présentations *AsciidoctorJ Reveal.js*.
Il expose un DSL minimal au consommateur et gère en interne toute la configuration des dépôts, dépendances et tâches.
Il intègre un pipeline RAG (Retrieval-Augmented Generation) à deux étapes pour la génération automatisée de decks AsciiDoc/Reveal.js via plusieurs fournisseurs LLM.

== Prérequis

* JDK 25 (testé avec Eclipse Temurin 25.0.2), support 23+
* Gradle 9.4.0+
* Docker (pour le container pgvector utilisé par le pipeline RAG)
* Node.js / npx (pour la tâche `serveSlides`)

== Décisions techniques clés

=== IDs de plugin — suffixe `.classic`
Depuis `asciidoctor-gradle-jvm-slides:5.0.0-alpha.1`, les IDs de plugin ont changé :

[cols="1,1"]
|===
| Ancien (4.x) | Actuel (5.x)

| `org.asciidoctor.jvm.gems`
| `org.asciidoctor.jvm.gems.classic`

| `org.asciidoctor.jvm.revealjs`
| `org.asciidoctor.jvm.revealjs.classic`
|===

=== Mode d'exécution — `OUT_OF_PROCESS`
`JAVA_EXEC` a été supprimé dans `5.0.0-alpha.1` suite aux changements de l'API Gradle sur la sérialisation des closures.
`OUT_OF_PROCESS` est le remplacement correct et fonctionne sur Gradle 9 + Java 25.

=== Dépôt pluginManagement — `repo.gradle.org`
La dépendance transitive `grolifant:0.12` (requise par `jruby-gradle-core-plugin`) n'est disponible que sur :

[source]
----
https://repo.gradle.org/gradle/libs-releases/
----
Ce dépôt doit être déclaré dans le `pluginManagement.repositories` du consommateur.

=== Dépôt Ivy pour les gems Ruby
La gem `asciidoctor-revealjs` est résolue via un dépôt Ivy dédié :
[source,kotlin]
----
project.repositories.ivy { repo ->
    repo.url = project.uri("https://rubygems.org/gems/")
    repo.patternLayout { layout -> layout.artifact("[module]-[revision].gem") }
    repo.metadataSources { s -> s.artifact() }
    repo.content { c -> c.includeGroup("rubygems") }
}
----
Le qualificateur `@gem` est obligatoire :
[source,kotlin]
----
project.dependencies.add("asciidoctorGems", "rubygems:asciidoctor-revealjs:3.1.0@gem")
----

=== PgVectorService — BuildService Gradle + docker-java

Le container pgvector est géré par `PgVectorService`, un `BuildService` Gradle à durée de vie
build-scoped. Il est lancé explicitement dans `start()` depuis l'action de tâche, jamais lors
de la phase de configuration.

Les tâches RAG déclarent la dépendance via `@ServiceReference` (Gradle 7.4+) — la seule
annotation qui garantit que Gradle maintient le service vivant pour toute la durée de la tâche.

[source,kotlin]
----
// RagTask.kt — @ServiceReference garantit le cycle de vie
abstract class RagTask : DefaultTask() {
    @get:ServiceReference
    abstract val pgVectorService: Property<PgVectorService>

    protected fun service(): PgVectorService =
        pgVectorService.get().also { it.start() }
}
----

Le port hôte est assigné dynamiquement (binding `0:5432`) pour éviter toute collision
avec un PostgreSQL existant. Le service est stoppé automatiquement par Gradle à la fin du build.

=== SSL pgvector — `sslmode=disable`

Le driver JDBC PostgreSQL tente une négociation SSL par défaut, ce qui provoque une
`EOFException` contre un container Docker sans SSL configuré. `RagManager.buildStore()`
crée un `PGSimpleDataSource` avec `setSsl(false)` / `setSslMode("disable")` (héritées de
`BaseDataSource`) et l'injecte via le constructeur protégé de `PgVectorEmbeddingStore`.

=== `--no-daemon` obligatoire pour les tâches RAG

Le daemon Gradle réutilise le process JVM entre les builds, empêchant le rechargement
de la lib native ONNX `libtokenizers.so` et provoquant une `UnsatisfiedLinkError` au
second build. Toutes les tâches RAG doivent être lancées avec `--no-daemon`, ou le projet
doit déclarer `org.gradle.daemon=false` dans `gradle.properties`.

=== Scaffold — `deck.properties` supprimé

`deck.properties` a été supprimé. La complétion de `slides/` est désormais vérifiée
par scan direct des fichiers `*-deck.adoc` dans `slides/misc/`. La fonction
`isSlidesConfigComplete` est une pure fonction Kotlin testable sans GradleTestKit.

[source,kotlin]
----
private fun isSlidesConfigComplete(miscDir: File): Boolean {
    if (!miscDir.resolve("index.html").exists()) return false
    return miscDir.listFiles { f ->
        f.isFile && f.name.endsWith("-deck.adoc")
    }?.isNotEmpty() ?: false
}
----

== SliderPlugin.apply() — Orchestration

[source,kotlin]
----
override fun apply(project: Project) {
    with(project) {
        checkJavaVersion()
        scaffoldSlidesIfAbsent()
        scaffoldSlidesContextIfAbsent()
        scaffoldDeckContextIfAbsent()
        configureRepositories()
        applyPlugins()
        configureDependencies()
        configureExtensions()
        registerTasks()
        afterEvaluate { createChatTasks() }  // enregistre toutes les tâches IA
    }
}
----

`createChatTasks()` est défini dans `AssistantManager` et enregistre :
- `PgVectorService` via `gradle.sharedServices.registerIfAbsent()`
- `reindexRag`, `proposeDeckContext`, `generateDeck` (tâches RAG typées)
- les tâches smoke-test `helloOllama*`, `helloGemini*`, `helloMistral*`, `helloHuggingFace*`

== Intégration IA — Pipeline RAG

`AssistantManager.createChatTasks()` est appelé dans `afterEvaluate` et enregistre
l'ensemble des tâches IA. Le pipeline RAG repose sur trois composants :

* `PgVectorService` — `BuildService` Gradle gérant le cycle de vie Docker via `docker-java`
* `RagManager` — indexation incrémentale par SHA-256, recherche par similarité cosinus
* `AssistantManager` — résolution du modèle, construction des prompts, appel LLM

=== Stratégie de prompt

`PromptManager` (objet interne de `AssistantManager`) gère deux prompts :

* `contextSystemPrompt` + `contextUserMessage` — étape 1 : propose un `DeckContext` JSON
* `deckSystemPrompt` + `deckUserMessage` — étape 2 : génère le deck AsciiDoc complet

Règles clés appliquées dans `deckSystemPrompt` :

* Maximum 5 points par slide avec `[%step]`, 7 sans
* Maximum 10 lignes de code par bloc `[source,...]`
* Jamais liste à puces et bloc de code sur la même slide
* `[NOTE.speaker]` et `[.notes]` contrôlés par `NotesConfiguration`
* `SlideHint` guide le contenu par slide sans remplacer la génération LLM

== Scaffold — Initialisation automatique

=== `scaffoldSlidesIfAbsent()`

Vérifie si le projet consommateur possède un dossier `slides/` complet. Si ce n'est pas le cas,
extrait le `slides.zip` embarqué dans le classpath du plugin.

Un dossier `slides/` est considéré complet quand `slides/misc/` contient :

* `index.html` — dashboard des présentations
* au moins un fichier `*-deck.adoc`

NOTE: `deck.properties` a été supprimé — les decks sont découverts par scan direct.

Le zip embarqué doit être placé à `slider/src/main/resources/slides.zip`.
Les entrées du zip doivent utiliser des chemins relatifs commençant par `slides/` :

[source]
----
slides/
└── misc/
    ├── example-deck.adoc
    └── index.html
----

=== `scaffoldSlidesContextIfAbsent()`

Si `slides-context.yml` est absent, génère automatiquement une instance `SlidesConfiguration`
par défaut sérialisée via `yamlMapper` — pas de YAML écrit en dur.

=== `scaffoldDeckContextIfAbsent()`

Si `slides/misc/example-deck-context.yml` est absent, génère un template `DeckContext`
prêt à l'emploi via `yamlMapper`.

Les trois fonctions sont des no-ops si la ressource existe déjà — le contenu existant du consommateur n'est jamais écrasé.
