== Description

`com.cheroliv.slider` est un plugin Gradle qui encapsule l'intégralité du cycle de vie de génération de présentations *AsciidoctorJ Reveal.js*.
Il expose un DSL minimal au consommateur et gère en interne toute la configuration des dépôts, dépendances et tâches.

== Version actuel : {plugin-version}

== Prérequis

* JDK 25 (testé avec Eclipse Temurin 25.0.2), support 23+
* Gradle 9.4.0+
* Node.js / npx (pour la tâche `serveSlides`)

== Décisions techniques clés

=== IDs de plugin — suffixe `.classic`
Depuis `asciidoctor-gradle-jvm-slides:5.0.0-alpha.1`, les IDs de plugin ont changé :

[cols="1,1"]
|===
| Ancien (4.x) | Actuel (5.x)

| `org.asciidoctor.jvm.gems`
| `org.asciidoctor.jvm.gems.classic`

| `org.asciidoctor.jvm.revealjs`
| `org.asciidoctor.jvm.revealjs.classic`
|===

=== Mode d'exécution — `OUT_OF_PROCESS`
`JAVA_EXEC` a été supprimé dans `5.0.0-alpha.1` suite aux changements de l'API Gradle sur la sérialisation des closures.
`OUT_OF_PROCESS` est le remplacement correct et fonctionne sur Gradle 9 + Java 25.

=== Dépôt pluginManagement — `repo.gradle.org`
La dépendance transitive `grolifant:0.12` (requise par `jruby-gradle-core-plugin`) n'est disponible que sur :

[source]
----
https://repo.gradle.org/gradle/libs-releases/
----
Ce dépôt doit être déclaré dans le `pluginManagement.repositories` du consommateur.

=== Dépôt Ivy pour les gems Ruby
La gem `asciidoctor-revealjs` est résolue via un dépôt Ivy dédié :
[source,kotlin]
----
project.repositories.ivy { repo ->
    repo.url = project.uri("https://rubygems.org/gems/")
    repo.patternLayout { layout -> layout.artifact("[module]-[revision].gem") }
    repo.metadataSources { s -> s.artifact() }
    repo.content { c -> c.includeGroup("rubygems") }
}
----
Le qualificateur `@gem` est obligatoire :
[source,kotlin]
----
project.dependencies.add("asciidoctorGems", "rubygems:asciidoctor-revealjs:3.1.0@gem")
----

=== Scaffold — Initialisation automatique à la première utilisation
À la première exécution d'une tâche Gradle, `SliderManager.Scaffold` effectue deux vérifications
indépendantes et génère automatiquement les ressources manquantes.

==== Initialisation de slides/ — `scaffoldSlidesIfAbsent()`
Vérifie si le projet consommateur possède un dossier `slides/` complet. Si ce n'est pas le cas,
extrait le `slides.zip` embarqué dans le classpath du plugin dans le répertoire du projet.

Un dossier `slides/` est considéré complet quand `slides/misc/` contient :

* `deck.properties` — au moins une clé `*.deck.file`
* `index.html` — dashboard des présentations
* au moins un fichier `*-deck.adoc` référencé par une clé `*.deck.file` dans `deck.properties`

Le zip embarqué doit être placé à :

[source]
----
slider/src/main/resources/slides.zip
----

Gradle l'intègre automatiquement dans le jar du plugin. Les entrées du zip doivent utiliser des chemins
relatifs commençant par `slides/` pour que l'extraction dans `projectDirectory` produise la bonne structure :

[source]
----
slides/
└── misc/
    ├── example-deck.adoc
    ├── index.html
    ├── deck.properties
    └── images/
----

La vérification de complétude est implémentée comme une fonction pure `isSlidesConfigComplete(miscDir: File)`
dans `SliderManager.Scaffold` — testable en pur Kotlin sans GradleTestKit.

==== Initialisation de slides-context.yml — `scaffoldSlidesContextIfAbsent()`
Si `slides-context.yml` est absent du répertoire du projet consommateur, en génère un automatiquement
en sérialisant une instance `SlidesConfiguration` par défaut via `yamlMapper`. Cela garantit que
la sortie est structurellement valide et cohérente avec le modèle de données — pas de chaîne YAML écrite en dur.

[source,kotlin]
----
val default = SlidesConfiguration(
    srcPath = "docs/asciidocRevealJs",
    pushSlides = GitPushConfiguration(
        from = "build/docs/asciidocRevealJs",
        to = "build/slides-repo",
        branch = "main",
        message = "deploy slides",
        repo = RepositoryConfiguration(
            name = "slides",
            repository = "https://github.com/your-org/your-slides-repo.git",
            credentials = RepositoryCredentials(
                username = "your-username",
                password = "your-token"
            )
        )
    )
)
yamlMapper.writeValue(slidesContext, default)
----

Les deux fonctions sont des no-ops si la ressource existe déjà — le contenu existant du consommateur n'est jamais écrasé.

== Build & Publication

=== Publier en local (pour les tests)

[source,bash]
----
./gradlew publishToMavenLocal
----

=== Lancer les tests

[source,bash]
----
# Tests unitaires
./gradlew test

# Tests fonctionnels (GradleTestKit)
./gradlew functionalTest

# Tests BDD Cucumber
./gradlew cucumberTest
----

=== Publier sur le Gradle Plugin Portal

[source,bash]
----
./gradlew publishPlugins
----
Nécessite `gradle.publish.key` et `gradle.publish.secret` dans `~/.gradle/gradle.properties`.

== DSL du plugin

[source,kotlin]
----
slider {
    // Chemin vers le fichier de configuration YAML (obligatoire)
    configPath = file("slides-context.yml").absolutePath
}
----

== Prérequis consommateur

La configuration consommateur est minimale — le plugin gère en interne toute la configuration des dépôts et dépendances.

=== settings.gradle.kts
[source,kotlin]
----
pluginManagement.repositories {
    mavenLocal()
    gradlePluginPortal()
}
rootProject.name = "votre-projet"
----

=== build.gradle.kts
[source,kotlin]
----
plugins { alias(libs.plugins.slider) }

slider {
    configPath = "slides-context.yml"
        .run(::file)
        .absolutePath
}
----

=== gradle/libs.versions.toml
[source,toml,subs="attributes+"]
----
[versions]
slider = "{plugin-version}"

[plugins]
slider = { id = "com.cheroliv.slider", version.ref = "slider" }
----

== Compatibilité des fonctionnalités Gradle

Le plugin déclare sa compatibilité avec les fonctionnalités opt-in de Gradle via le bloc
`compatibility {}` introduit dans Plugin Publish plugin v2.1.0.
Cette déclaration est obligatoire — le Gradle Plugin Portal émet un avertissement
lors du `publishPlugins` si elle est absente, et prévoit de rejeter les plugins sans déclaration.

[source,kotlin]
----
import org.gradle.plugin.compatibility.compatibility

gradlePlugin {
    plugins {
        create("slider") {
            // ...
            compatibility {
                features {
                    // asciidoctorRevealJs tourne OUT_OF_PROCESS via JRuby —
                    // incompatible avec le Configuration Cache.
                    // Sera réévalué quand asciidoctor-gradle se stabilise
                    // au-delà de 5.0.0-alpha.1.
                    configurationCache = false
                }
            }
        }
    }
}
----

Déclarer `configurationCache = false` est honnête et recommandé — cela informe
clairement les utilisateurs et n'a pas de conséquences négatives sur le classement
du plugin sur le Portal au-delà du badge Configuration Cache.

== Roadmap
* Support du Configuration Cache — bloqué sur la version stable d'asciidoctor-gradle `5.x`.
* DSL étendu — thème, transition, répertoire source configurables.
* Tests fonctionnels automatisés couvrant l'intégralité du cycle de build.
